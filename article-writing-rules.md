# ppu.jp 記事執筆ルール（article-writing-rules）

> v1.0 / 2026-06-13
> 土台: [stop-slop](https://github.com/hardikpandya/stop-slop)（MIT, hardikpandya）を日本語SEO＋E-E-A-T向けに翻案
> **運用: 全記事の新規生成・改稿時に本ルールへ適合させること（必須参照）。** §5チェックリストで自己採点し、合格基準を満たすまで公開しない。
> 関連: `gatto-scope.md` `content-calendar.md`（ゴールデンテンプレート）

## 0. 前提となる決定（2026-06-13）

| 項目 | 方針 |
|---|---|
| E-E-A-T | **実在の監修者を全記事に適用**（角間 実／株式会社エコノミア代表）。架空・盛りNG |
| 一次情報 | **「待たない設計」。記事ごとの一次"体験"は必須にしない**。代わりに §2 の3本柱で担保 |
| 作例画像 | 「こう撮れます」の作例は実写真 or 出典付き。**AI画像はヒーロー・抽象イメージ・装飾のみ**。無理に作例を作らない |
| 文体 | **バランス適用**（SEO可読性は残しつつ明確なAI tellを除去） |
| 運用 | リポジトリ内ファイル＋生成時必須参照 |

## 1. 禁止パターン（AIっぽさの除去 / stop-slop翻案）

日本語記事に効くものを優先。各項目 **兆候 → 直し方**。

### 文字装飾・記号
- **太字の乱打** → 1セクションあたり1〜2箇所まで。要点のみ。
- **全角ダッシュ「——」の多用** → 句点・読点に置換。原則使わない。
- **三点リストの常用**（「正確・魅力的・…」の三連、体言止め三連） → 項目数を2や4に崩す、または散文化。

### 言い回し
- **throat-clearing（前置き）**：「この記事では〜を解説します」「結論から言うと」「まず〜について」 → 削除し、いきなり要点へ。
- **vague declaratives（中身のない断定）**：「〜と言っても過言ではない」「〜が9割」「重要なのは」「〜は奥が深い」 → 具体・数値・出典に置換。
- **binary contrast / negative listing**：「Xではない。Yだ」「単なるXではなく、Yである」 → 否定の助走を外し、Yを直接言う。
- **false agency（無生物主語の擬人化）**：「データが教えてくれる」「写真が売上を決める」「課題が改善につながる」 → 主語を人に（「購入者は写真で判断する」）。
- **meta-commentary**：「次のセクションでは」「見ていきましょう」「ここからが本題」 → 削除。
- **過剰な強調語**：「とても」「非常に」「まさに」「圧倒的」「徹底的に」 → 削るか具体語へ。

### リズム
- **語尾の単調**：「〜できます。〜しましょう。〜です。」の連打 → 語尾と文長を混ぜる。
- **段落の締めを毎回パンチラインにしない**。
- **体言止めの乱用**を避ける（時々なら可）。

## 2. E-E-A-T（待たない設計：一次"体験"に依存しない3本柱）

SEO没個性リスク対策の核。**一次情報の収集を待つと量産が止まる**ため、記事ごとの実体験には依存せず、次の3本柱で担保する。

### 柱1: 専門家による監修（記事の主題で使い分け）
全記事に監修ボックス（`SupervisorBox`）を表示し、schema に `reviewedBy`(Person) を付与する。監修者は **3名** から記事の主題に合わせて選ぶ（frontmatter `supervisor: kakuma | tsukada | hinata`、未指定はデフォルト＝角間）。データは `src/config/experts.ts`。

| ID | 監修者 | 専門 | 割り当てる記事の例 |
|---|---|---|---|
| `kakuma` | 角間 実（株式会社エコノミア 代表取締役社長） | EC・D2C戦略／売れる商品写真の設計 | EC・メルカリ・費用・売れる商品写真・D2C・Amazon/楽天 |
| `tsukada` | 塚田 栄一（塚田スタジオ 執行役員・カメラマン） | 化粧品・美容・医療の撮影／撮影技術全般 | 化粧品・コスメ・美容・サプリ・医療＋ライティング・俯瞰・背景・撮影ボックス・カメラ/レンズ・指輪などの撮影技術 |
| `hinata` | 日向 正樹（塚田スタジオ 制作部統括） | 撮影の進行管理・品質管理 | 撮影会社の選び方・発注の流れ・品質チェック・大量撮影の進行管理 |

- これで各記事に Expertise / Authoritativeness を付与する（個別の一次体験が無くてもE-E-A-Tが立つ）。
- ⚠️ **実在情報のみ**。肩書き・実績を盛らない。実写真が無い間はイニシャル表示（実在人物のAI生成顔写真は使わない）。

### 柱2: 出典付きの外部データ
- メーカー画像要件・ECモール（Amazon/楽天等）の公式規定・公的データは **一次ソースにリンク**。
- 憶測の数字（「20〜30%向上」「○○が9割」等）を **出典なしで書かない**。書けないなら数字を出さず定性表現に。

### 柱3: 比較メディアとしての独自集約
- 46社の料金・特徴・比較表など、**ppu.jpが集約した情報そのものが独自価値**。各記事から該当する比較ハブ・会社ページへ内部リンクし、「情報を集めて比較できる」価値を渡す。

### やらないこと（無い設計の境界）
- **実体験を装わない**：「私が撮影した」「自社調査で」など、取得していない一次体験・数値を書かない。
- **AI画像を作例にしない**：「こう撮れます」の作例は実写真 or 出典付きのみ。AI画像はヒーロー・抽象イメージ・装飾に限定し、作例と誤認させるキャプションを付けない。
- 困ったら断定せず、出典 or 定性表現に逃がす。

## 3. 文体（バランス適用）

- です・ます調。読者は二人称「**あなた**」（「ユーザーは」「人は」より優先）。
- **見出し直下の要点太字（1〜2箇所）と箇条書きは許容**。ただし説明の本体は散文を基本に。
- 文の長短を混ぜる。1段落＝1論点。
- 専門用語は初出で一言説明（例: 物撮り＝モノを被写体にした撮影）。
- 読者像: EC事業者・メーカー担当・個人出品者。煽らず実務的に。

---

## 4. SEO要件（ゴールデンテンプレ準拠 / 抜粋）

- `title`: 主要KWを前方＋年号 / `description`: 120字前後 / 導入100字に主要KW・結論。
- frontmatter: `faq`(3〜5) / 手順系は `howto` / `relatedSlugs` / `heroImage`。
- 内部リンク3層: 商用ハブ1＋関連スポーク2〜3＋関連社（タグ自動）。
- 文字数目安: スポーク2,500〜4,000字 / ハブ3,000〜5,000字。
- 主要KWは title / h1 / 導入 / H2の1つ / description に配置（過剰最適化はしない）。

## 5. 公開前チェックリスト

**AIっぽさ（§1）**
- [ ] 太字は各セクション1〜2箇所以内
- [ ] 「——」を使っていない
- [ ] 三点リストの連打がない
- [ ] 前置き（throat-clearing）で始まっていない
- [ ] 中身のない断定（「〜が9割」等）がない
- [ ] 無生物主語の擬人化がない
- [ ] 語尾・文長に変化がある

**E-E-A-T（§2 / 待たない設計）**
- [ ] 監修ボックス（角間 実）が表示され schema `reviewedBy` 付与
- [ ] 出典なしの数字・断定がない（数値は一次ソースにリンク）
- [ ] 比較ハブ・会社ページへ内部リンク（独自集約の価値を渡す）
- [ ] 実体験を装っていない（未取得の体験・自社数値なし）
- [ ] AI画像を作例として使っていない

**SEO（§4）**
- [ ] KW配置OK / 内部リンク3層 / FAQ・schema 設定済み

## 6. 採点ルーブリック（stop-slop準拠）

5軸を各10点で採点。**合計35/50未満はリライト**。本プロジェクトは ④authenticity を重視。

1. directness（直接性・前置きの少なさ）
2. rhythm（文長・語尾の多様性）
3. trust（出典・誠実さ）
4. **authenticity（一次情報・監修・固有性）← 重み付け**
5. density（情報密度・冗長さ）

## 7. SEO臭さ消去の判定基準（機械検出＋しきい値）

> v1.1追加（2026-06-14）。§1の禁止パターンを **grepで検出できる客観しきい値** に落とし込んだもの。記事ごとに下表を機械測定し、1つでも🔴に触れたら「中以上」＝**要リライト**。stop-slopの5軸（§6）はリライト後の最終判定に使う。

### 7-1. 検出シグナルとしきい値（1記事あたり、本文のみ）

| シグナル | 検出（概念） | 🟢低 | 🟡中 | 🔴高 |
|---|---|---|---|---|
| **数字見出しlisticle** | `## ◯つのポイント/ステップ/コツ/理由` 形式の見出し | 0 | 1 | 2+ |
| **太字ラベル足場** | 行頭 `**確認すべきこと:**` の直後に箇条書き、を反復 | 0 | 1〜2 | 3+ |
| **太字の総数** | 本文中の `**…**` 数 | ≤1×H2数 | 〜2×H2数 | それ超 |
| **半角スペース崩れ** | `EC サイト` `3〜5 カット` 等、英数字↔和文間のスペース | 0 | 1〜2 | 3+ |
| **出典なし成果断定** | 「向上します／高まります／直結します／決定づけ／9割／千差万別／大きく左右」 | 0 | 1 | 2+ |
| **タイトル過剰修飾** | 「徹底解説」「完全版」「大全」の重ね（※年号【20xx】は§4の設計、残してよい） | 0 | 1 | 2+ |
| **前置き/メタ** | 「この記事では」「順に見ていきます」「結論から言うと」 | 0 | 1 | 2+ |
| **汎用署名のみ** | `author: 編集部` だけで `supervisor` 未設定 | supervisorあり | — | supervisorなし |

### 7-2. 総合判定（リライトの線引き）

- 🔴が **1つでもあれば「高」＝要リライト**。
- 🟡が **2つ以上で「中」＝要リライト**。
- 🟡が1つ以下かつ🔴ゼロ＝「低」＝据え置き可（タイトル修飾・署名だけ軽微修正でも可）。
- **「中以上＝全リライト」が本プロジェクトの運用方針（2026-06-14）。**

### 7-3. リライト時の置換ルール（兆候→直し）

- 数字見出しlisticle → 散文の小見出し（「白背景で清潔感を出す」等、内容を表す見出し）に。順序が意味を持つ手順だけ番号を残す。
- 太字ラベル足場（`**確認すべきこと:**`＋箇条書き）→ ラベルを削り、リード文＋箇条書き、または散文化。太字はH2あたり1〜2に圧縮。
- 半角スペース崩れ → 詰める（`ECサイト` `3〜5カット`）。表記ゆれ（全角/半角）も統一。
- 出典なし成果断定 → 一次ソースにリンク or 定性表現（「伝わりやすくなる」程度）に後退。盛らない。
- タイトル過剰修飾 → 主要KW＋年号は残し、「徹底解説」等の空語を削る。
- 前置き/メタ → 削除して要点から始める。

## 8. 最終調査基準（公開ゲート）

> リライト後に必ず通す最終レビュー。**全項目PASSで初めて公開**。1つでもFAILなら戻す。

1. **機械再測定（§7）**：リライト後に§7-1を再grepし、🔴ゼロ・🟡1以下であること。
2. **stop-slop 5軸採点（§6）**：directness / rhythm / trust / **authenticity（重み）** / density を各10点。**合計40/50以上**（本プロジェクトは§6の35より厳格化）。authenticityが6未満なら不合格。
3. **E-E-A-Tゲート（§2）**：監修ボックス表示＋schema `reviewedBy`、出典なし数値ゼロ、比較ハブ/会社ページへの内部リンク3層、実体験を装っていない、AI画像を作例にしていない。
4. **事実・リンク健全性**：内部リンク先が実在（404でない）、KW配置（title/h1/導入/H2/description）、FAQ・howto・schemaが壊れていない。
5. **音読チェック**：冒頭3文を声に出して読み、前置き・擬人化・語尾単調が無いこと（人間の編集者が最後に1回）。

合格記録は本ファイルかPR説明に「§8 PASS（5軸合計◯/50）」と残す。
